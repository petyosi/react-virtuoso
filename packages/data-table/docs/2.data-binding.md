Every data table starts with one data source. The column definitions decide what is visible, but the
data binding decides where rows come from and who owns changes to filtering, sorting, grouping, and
loading.

Use one of these bindings:

- `data` for plain React state or already prepared table data.
- `localSource()` when the browser owns derived views such as filtering, sorting, and grouping
- `remoteSource()` when the viewport, query state, or backend owns the row set.

## Direct data

Pass `data={{ data, groups: [] }}` when you already have the rows in memory.

```tsx live
import { DataTable, DataTableCell, DataTableColumn, DataTableColumnHeader } from '@/components/ui/data-table'

const rows = [
  { id: 'SKU-001', name: 'Standing Desk', category: 'Office', stock: 14 },
  { id: 'SKU-002', name: 'USB-C Dock', category: 'Peripherals', stock: 42 },
  { id: 'SKU-003', name: 'Mechanical Keyboard', category: 'Peripherals', stock: 28 },
]

export default function App() {
  return (
    <DataTable className="rounded-xl" data={{ data: rows, groups: [] }} style={{ height: 280 }}>
      <DataTableColumn field="name">
        <DataTableColumnHeader>Product</DataTableColumnHeader>
        <DataTableCell className="font-medium">{({ cellValue }) => String(cellValue)}</DataTableCell>
      </DataTableColumn>
      <DataTableColumn field="category">
        <DataTableColumnHeader>Category</DataTableColumnHeader>
        <DataTableCell>{({ cellValue }) => String(cellValue)}</DataTableCell>
      </DataTableColumn>
      <DataTableColumn field="stock">
        <DataTableColumnHeader className="justify-end">Stock</DataTableColumnHeader>
        <DataTableCell className="text-right tabular-nums">{({ cellValue }) => String(cellValue)}</DataTableCell>
      </DataTableColumn>
    </DataTable>
  )
}
```

## Local data model

Use `localSource()` when the source rows stay in the browser but the visible rows are derived through
actions. A local model has two different kinds of actions:

- pipeline actions derive the currently visible rows from the source rows
- source mutators change the backing source array, then recompute the current view

```tsx
const model = localSource({
  data: products,
  pipeline: ['filter', 'sort'],
  actions: {
    filter: {
      stage: 'filter',
      handler: ({ data, payload }) => data.filter((row) => row.category === payload),
    },
    sort: {
      stage: 'sort',
      handler: ({ data }) => data.toSorted((a, b) => a.name.localeCompare(b.name)),
    },
    renameProduct: {
      handler: ({ source, payload }) => source.map((row) => (row.id === payload.id ? { ...row, name: payload.name } : row)),
    },
  },
})

model.send({ action: 'filter', payload: 'Office' })
model.send({ action: 'renameProduct', payload: { id: 'SKU-001', name: 'Desk Pro' } })
```

Bind the model with `model={model}` instead of the `data` prop. A local pipeline stage can return
either a flat row array or `{ data, groups }`.

Pipeline stages are cached per table view. When a pipeline action changes the payload for one stage,
that stage and every later stage rerun; earlier cached stages are reused. A source mutator invalidates
all stages because the backing array changed.

Use `model.setData(nextRows, nextGroups)` when a local source receives a fresh dataset from React
state, a websocket, or a parent workflow. Passing `nextGroups` is optional; omit it to keep the
current group metadata.

## Local action persistence

Pipeline actions can opt into persistence so `modelStatePersistenceAdapter()` can save filter, sort,
and grouping state.

```tsx
const model = localSource({
  data: products,
  pipeline: ['filter'],
  actions: {
    filter: {
      stage: 'filter',
      persistence: {
        isEmpty: (value) => !value,
      },
      handler: ({ data, payload }) => (payload ? data.filter((row) => row.category === payload) : data),
    },
  },
})
```

Set `persistence: true` to store the action payload as-is. Use an object when the persisted shape
needs a custom `key`, `capture`, `restore`, or `isEmpty` function.

## Remote data model

Use `remoteSource()` when the table should ask for the rows that match the current viewport and query
state.

```tsx
const model = remoteSource({
  fetch: fetchProducts,
  initialParams: { category: 'all' },
  onViewportChange: defaultOffsetViewportHandler,
  pageSize: 50,
  placeholder: loadingProduct,
  actions: {
    filter: {
      handler: ({ params, payload }) => ({ ...params, category: payload as string }),
    },
  },
})
```

Offset mode fetches a slice from a known total count. Append mode (`mode: 'append'`) works better for
cursor-style feeds and loads more rows as the user reaches the end.

## Remote actions

Remote actions transform application payloads into query params, then trigger a refresh with the new
params.

```tsx
const model = remoteSource({
  fetch: fetchProducts,
  initialParams: { category: 'all', search: '' },
  actions: {
    search: {
      persistence: true,
      handler: ({ params, payload }) => ({ ...params, search: String(payload) }),
    },
    category: {
      persistence: { key: 'categoryFilter' },
      handler: ({ params, payload }) => ({ ...params, category: String(payload) }),
    },
  },
})

model.send({ action: 'search', payload: 'desk' })
```

The action handler receives the current params and returns the next params. Persisted remote actions
store the params value after the handler runs, so restored state can rebuild the same request.

## Concurrency and cancellation

Data model actions can set `strategy`:

- `supersede` cancels the active request for the same action and uses the latest payload.
- `queue` runs requests in order.
- `deduplicate` ignores a duplicate in-flight payload.

Remote `fetch` functions receive an `AbortSignal`. Pass it to `fetch()` or your client library so a
superseded refresh, viewport request, or disconnect can cancel work cleanly.

```tsx
async function fetchProducts({ offset, limit, params, signal }) {
  const response = await fetch(`/api/products?offset=${offset}&limit=${limit}&q=${params.search}`, {
    signal,
  })
  return response.json()
}
```

`onError` is called for failed fetches. Aborted requests publish a loading cancellation event instead
of an error.

## Viewport handlers

`onViewportChange` decides what a remote source should load when the rendered range changes.
`defaultOffsetViewportHandler` requests missing pages around the current range. `defaultAppendViewportHandler`
requests more rows when the end of an append feed approaches.

Use a custom handler when your backend has different paging boundaries, prefetch rules, or cache
requirements.

## Group metadata

Grouped rows are still part of the data binding. The table data shape is always:

```ts
{
  data: rowsAndGroupRows,
  groups: [{ index: 0, level: 0 }]
}
```

For direct data, pass that shape to `data`. For local models, return that shape from a pipeline stage
or call `model.setData(nextRows, nextGroups)`. For remote models, return `groups` from the fetch
result. See [Grouped Rows](/data-table/features/grouped-rows/) for the full grouped-row contract.

## Loading state

`remoteSource()` publishes semantic loading state for initial loads, refreshes, and append-mode end
loads. Treat the loading UI as visual customization: replace `components.LoadingPlaceholder`,
`components.LoadingOverlay`, and `components.LoadingFooter` in the table wrapper when the default
shadcn states do not match your product.

The same loading lifecycle is also available through the model event channel:

```tsx
const unsubscribe = model.subscribe((message) => {
  if (message.type === 'event' && message.payload?.kind === 'loading') {
    console.log(message.payload.reason, message.payload.phase)
  }
})
```

The table bridge consumes those events and updates `loadingState$`, which is what
`useVirtuosoLoadingState()` and the loading slots read.
