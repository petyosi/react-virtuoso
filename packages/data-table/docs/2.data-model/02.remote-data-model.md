Use `remoteModel()` when the table should request rows instead of receiving a complete in-memory
array. The model turns table events and application actions into fetch requests, cancels stale work,
and publishes loading events and returned rows back to the table.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  initialParams: { category: 'all', search: '' },
  onViewportChange: defaultOffsetViewportHandler,
  pageSize: 50,
  placeholder: loadingProduct,
})
```

## Choose a fetch mode

Pick the mode that matches your backend.

| Backend shape | Use | Why |
| --- | --- | --- |
| The API accepts `offset` and `limit` and returns a total count | `defaultOffsetViewportHandler` | The table can request missing slices around the currently rendered rows and preserve scroll height with placeholders |
| The API returns the next cursor or page token | `mode: 'append'` with `defaultAppendViewportHandler` | The table grows as the user reaches the end of the loaded rows |
| The API uses offsets but needs a different prefetch window | custom `onViewportChange` | Your handler can choose exact `{ offset, limit }` slices from the rendered range |

Offset mode is usually the right fit for large tabular result sets. Append mode is better for feeds,
logs, and "load more" APIs where arbitrary jumps are not available.

## Fetch results

Offset mode fetches a slice from a known total count.

```typescript
async function fetchProducts({ offset, limit, params, signal }) {
  const response = await fetch(`/api/products?offset=${offset}&limit=${limit}&search=${params.search}`, {
    signal,
  })

  return response.json()
}
```

The fetch result should include `rows` and `totalCount`. Return `groups` when the backend gives the
table a flattened grouped result.

```typescript
return {
  rows: result.rows,
  groups: result.groups,
  totalCount: result.totalCount,
}
```

When a remote endpoint defines the row schema, return schema metadata beside the rows or keep a
separate field list in your application state. See
[Columns for Dynamic Schemas](/data-table/columns/inferring-columns-from-model-data/) for
dynamic remote schemas and placeholder rows.

`pageSize` is the requested slice size. Use a value large enough that ordinary scrolling does not
trigger a request on every small movement, but small enough that refreshes stay responsive.

`placeholder` is the row shape used while the first offset pages are pending. Include every field
that your column declarations read, especially when columns are generated from remote schema data.

## Remote actions

Remote actions translate UI intent into the params for the next request. URL parameters are a common
case, but the same pattern works for cursor filters, request bodies, search objects, or any other
input your fetch function needs.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  initialParams: { category: 'all', search: '' },
  actions: {
    search: {
      strategy: 'supersede',
      persistence: true,
      handler: ({ params, payload }) => ({ ...params, search: String(payload) }),
    },
    category: {
      persistence: { key: 'categoryFilter' },
      handler: ({ params, payload }) => ({ ...params, category: String(payload) }),
    },
  },
})

model.send({ action: 'search', payload: 'desk' })
```

The action handler receives the current params and the action payload, then returns the next params.
Returning params triggers a refresh. If an action opts into persistence, the persistence adapter
stores the action payload by default and replays the handler from `initialParams` when state is
restored. Use `capture` and `restore` when you need a custom serialized shape.

## Concurrency and cancellation

Use action strategies when users can send actions faster than the backend can answer.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  actions: {
    search: {
      strategy: 'supersede',
      handler: ({ params, payload }) => ({ ...params, search: String(payload) }),
    },
    exportRows: {
      strategy: 'queue',
      handler: ({ params, payload }) => ({ ...params, exportId: String(payload) }),
    },
  },
})
```

- `supersede` is best for search boxes and filters. It cancels the active request for the same
  action and uses the latest payload.
- `queue` is best for ordered workflows where every request must run.
- `deduplicate` is best for repeated identical commands, such as retry buttons or repeated load
  requests.

Remote `fetch` functions receive an `AbortSignal`. Pass it to `fetch()` or your client library so a
superseded refresh, viewport request, or disconnect can cancel work cleanly.

```typescript
async function fetchProducts({ offset, limit, params, signal }) {
  const response = await fetch(`/api/products?offset=${offset}&limit=${limit}&q=${params.search}`, {
    signal,
  })
  return response.json()
}
```

`onError` is called for failed fetches. Aborted requests publish a loading cancellation event instead
of an error.

## Viewport handlers

`onViewportChange` decides what the model should fetch when the rendered row range changes. The
default handlers cover the common offset and append cases; custom handlers are for APIs whose fetch
window does not match those shapes.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  onViewportChange: defaultOffsetViewportHandler,
  pageSize: 50,
})
```

`defaultOffsetViewportHandler` requests missing pages around the currently rendered rows. Use it for
paginated APIs that accept `offset` and `limit`.

Append mode loads more rows as the user approaches the end of the loaded list.

```typescript
const model = remoteModel({
  mode: 'append',
  fetch: fetchFeed,
  onViewportChange: defaultAppendViewportHandler,
  pageSize: 30,
})
```

Use append mode for cursor-style feeds, activity streams, and "load more" APIs. Use a custom
viewport handler when offset mode should prefetch a different window than the built-in handler.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  onViewportChange: ({ endIndex, pageSize, totalCount }) => {
    const nextOffset = Math.floor((endIndex + 1) / pageSize) * pageSize

    if (nextOffset < totalCount) {
      return {
        fetch: [{ offset: nextOffset, limit: Math.min(pageSize, totalCount - nextOffset) }],
      }
    }
  },
  pageSize: 50,
})
```

The returned action describes which rows the model should request. Append mode can request the next
page with `loadMore`; offset mode can request one or more missing `{ offset, limit }` slices. The
built-in handlers calculate those requests for the common cases.

## Loading state

The table consumes remote loading events automatically. The shadcn wrapper already displays default
loading placeholder, overlay, and footer states. You only need subscriptions when surrounding UI
outside the table must react to the same lifecycle.

The loading state separates the user-visible cases:

- `initial` for the first unresolved dataset
- `refresh` when request params change while old rows stay visible
- `end` when append mode fetches more rows below the current data

Offset viewport fetches usually appear as placeholder rows instead of a table-level loading overlay,
so they do not set `loadingState$`. Manual `loadRange` requests emit `viewport` events for
subscribers, but the loading slots ignore that reason.

Replace loading UI through table components when the defaults do not match your product.

```tsx
function LoadingOverlay({ loadingState }) {
  if (loadingState.refresh.status === 'error') {
    return <div role="status">Refresh failed: {loadingState.refresh.errorMessage}</div>
  }

  return <div role="status">Refreshing rows...</div>
}

;<DataTable components={{ LoadingOverlay }} model={model} />
```

Use the event channel for external controls, logging, or diagnostics around remote fetches.

```typescript
const unsubscribe = model.subscribe((message) => {
  if (message.type === 'event' && message.payload?.kind === 'loading') {
    console.log(message.payload.reason, message.payload.phase)
  }
})
```

The table bridge consumes `initial`, `refresh`, and `end` events and updates `loadingState$`, which
is what `useVirtuosoLoadingState()` and the loading slots read.
