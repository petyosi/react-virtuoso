Use `remoteModel()` when the table should request rows instead of receiving a complete in-memory
array. The model owns fetch scheduling, cancellation, loading events, and query state.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  initialParams: { category: 'all', search: '' },
  onViewportChange: defaultOffsetViewportHandler,
  pageSize: 50,
  placeholder: loadingProduct,
})
```

## Choose a loading mode

Pick the mode that matches your backend.

| Backend shape | Use | Why |
| --- | --- | --- |
| The API accepts `offset` and `limit` and returns a total count | `defaultOffsetViewportHandler` | The table can request missing slices around the current viewport and preserve scroll height with placeholders |
| The API returns the next cursor or page token | `mode: 'append'` with `defaultAppendViewportHandler` | The table grows as the user reaches the end of the loaded rows |
| The API can load previous and next windows around a cursor | custom `onViewportChange` | Your handler can request earlier or later rows when the rendered range approaches either edge |

Offset mode is the default mental model for large tabular result sets. Append mode is better for
feeds, logs, and "load more" APIs where arbitrary jumps are not available.

## Fetch results

Offset mode fetches a slice from a known total count.

```typescript
async function fetchProducts({ offset, limit, params, signal }) {
  const response = await fetch(`/api/products?offset=${offset}&limit=${limit}&search=${params.search}`, {
    signal,
  })

  return response.json()
}
```

The fetch result should include `rows` and `totalCount`. Return `groups` when the backend gives the
table a flattened grouped result.

```typescript
return {
  rows: result.rows,
  groups: result.groups,
  totalCount: result.totalCount,
}
```

When a remote endpoint owns the row schema, return schema metadata beside the rows or keep a
separate field list in your application state. See
[Columns for Dynamic Schemas](/data-table/columns/inferring-columns-from-model-data/) for
dynamic remote schemas and placeholder rows.

`pageSize` is the requested slice size. Use a value large enough that ordinary scrolling does not
trigger a request on every small movement, but small enough that refreshes stay responsive.

`placeholder` is the row shape used while the first offset pages are pending. Include every field
that your column declarations read, especially when columns are generated from remote schema data.

## Remote actions

Remote actions translate UI intent into the next remote request state. Query params are the common
case, but the same pattern works for cursor filters, request bodies, search objects, or any state
your fetch function needs.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  initialParams: { category: 'all', search: '' },
  actions: {
    search: {
      strategy: 'supersede',
      persistence: true,
      handler: ({ params, payload }) => ({ ...params, search: String(payload) }),
    },
    category: {
      persistence: { key: 'categoryFilter' },
      handler: ({ params, payload }) => ({ ...params, category: String(payload) }),
    },
  },
})

model.send({ action: 'search', payload: 'desk' })
```

The action handler receives the current params and returns the next params. Returning new params
triggers a refresh. Persisted remote actions store the params value after the handler runs, so a
restored table can rebuild the same request.

## Concurrency and cancellation

Use action strategies when users can send actions faster than the backend can answer.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  actions: {
    search: {
      strategy: 'supersede',
      handler: ({ params, payload }) => ({ ...params, search: String(payload) }),
    },
    exportRows: {
      strategy: 'queue',
      handler: ({ params, payload }) => ({ ...params, exportId: String(payload) }),
    },
  },
})
```

- `supersede` is best for search boxes and filters. It cancels the active request for the same
  action and uses the latest payload.
- `queue` is best for ordered workflows where every request must run.
- `deduplicate` is best for repeated identical commands, such as retry buttons or repeated load
  requests.

Remote `fetch` functions receive an `AbortSignal`. Pass it to `fetch()` or your client library so a
superseded refresh, viewport request, or disconnect can cancel work cleanly.

```typescript
async function fetchProducts({ offset, limit, params, signal }) {
  const response = await fetch(`/api/products?offset=${offset}&limit=${limit}&q=${params.search}`, {
    signal,
  })
  return response.json()
}
```

`onError` is called for failed fetches. Aborted requests publish a loading cancellation event instead
of an error.

## Viewport handlers

`onViewportChange` decides what the model should load when the rendered range changes. The default
handlers cover the common offset and append cases; custom handlers are for APIs whose loading window
does not match those shapes.

```typescript
const model = remoteModel({
  fetch: fetchProducts,
  onViewportChange: defaultOffsetViewportHandler,
  pageSize: 50,
})
```

`defaultOffsetViewportHandler` requests missing pages around the current viewport. Use it for
paginated APIs that accept `offset` and `limit`.

Append mode loads more rows as the user approaches the end of the loaded list.

```typescript
const model = remoteModel({
  mode: 'append',
  fetch: fetchFeed,
  onViewportChange: defaultAppendViewportHandler,
  pageSize: 30,
})
```

Use append mode for cursor-style feeds, activity streams, and "load more" APIs. If your API can load
earlier rows too, use a custom viewport handler that requests previous cursors when the rendered
range approaches the start of the loaded window.

```typescript
const model = remoteModel({
  fetch: fetchMessages,
  onViewportChange: ({ range, dispatch }) => {
    if (range.startIndex < 10) {
      dispatch({ type: 'start' })
    }

    if (range.endIndex > range.totalCount - 10) {
      dispatch({ type: 'end' })
    }
  },
})
```

The `dispatch` calls describe which side of the loaded window needs more data. Use `type: 'start'`
for previous rows and `type: 'end'` for following rows. In offset mode, the built-in handler
calculates the missing pages for you.

## Loading state

The table consumes remote loading events automatically. The shadcn wrapper already displays default
loading placeholder, overlay, and footer states. You only need subscriptions when surrounding UI
outside the table must react to the same lifecycle.

The loading state separates the user-visible cases:

- `initial` for the first unresolved dataset
- `refresh` when query params change while old rows stay visible
- `end` when append mode fetches more rows below the current data
- `start` when a custom handler fetches rows before the current data

Replace loading UI through table components when the defaults do not match your product.

```tsx
function LoadingOverlay({ loadingState }) {
  if (loadingState.refresh.status === 'error') {
    return <div role="status">Refresh failed: {loadingState.refresh.errorMessage}</div>
  }

  return <div role="status">Refreshing rows...</div>
}

;<DataTable components={{ LoadingOverlay }} model={model} />
```

Use the event channel for external controls, logging, or diagnostics.

```typescript
const unsubscribe = model.subscribe((message) => {
  if (message.type === 'event' && message.payload?.kind === 'loading') {
    console.log(message.payload.reason, message.payload.phase)
  }
})
```

The table bridge consumes those events and updates `loadingState$`, which is what
`useVirtuosoLoadingState()` and the loading slots read.
