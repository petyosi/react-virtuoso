The table does not infer columns automatically. Columns are React declarations so you can control
labels, formatting, visibility, ordering, persistence, and interactive header behavior. When the row
schema is dynamic, discover or receive a field list in your app and map it to `DataTableColumn`
declarations.

This feature crosses the data model boundary. Local models can discover fields from rows already in
memory. Remote models often get the field list from the backend, from the first resolved page, or
from placeholder rows used before the first request completes.

## Prefer explicit schema metadata

When the backend knows the shape, return column metadata beside the rows. This gives the table
stable labels, order, type hints, and visibility before the first body row is rendered.

```typescript
type ProductColumn = {
  field: string
  label: string
  align?: 'start' | 'end'
}

type ProductResponse = {
  rows: Record<string, unknown>[]
  totalCount: number
  columns: ProductColumn[]
}
```

Render the metadata as column declarations.

```tsx
{
  columns.map((column) => (
    <DataTableColumn field={column.field} key={column.field}>
      <DataTableColumnHeader className={column.align === 'end' ? 'justify-end' : undefined}>{column.label}</DataTableColumnHeader>
      <DataTableCell className={column.align === 'end' ? 'text-right tabular-nums' : undefined}>
        {({ cellValue }) => String(cellValue ?? '')}
      </DataTableCell>
    </DataTableColumn>
  ))
}
```

Explicit metadata is usually the best remote-data approach because the server can keep the schema
stable across filters, pagination, permissions, and empty result sets.

## Local model fields

For local models, derive fields from a representative set of source rows and keep the order stable.

```tsx
const fields = React.useMemo(() => {
  const seen = new Set<string>()
  const next: string[] = []

  for (const row of rows) {
    for (const field of Object.keys(row)) {
      if (field !== 'id' && !seen.has(field)) {
        seen.add(field)
        next.push(field)
      }
    }
  }

  return next
}, [rows])
```

A complete local example can derive fields and pass the same rows to `localModel()`.

```tsx live
import * as React from 'react'

import { DataTable, DataTableCell, DataTableColumn, DataTableColumnHeader } from '@/components/ui/data-table'
import { localModel } from '@virtuoso.dev/data-table'

const rows = [
  { id: 'SKU-001', product: 'Standing Desk', category: 'Office', stock: 14 },
  { id: 'SKU-002', product: 'USB-C Dock', category: 'Peripherals', stock: 42, region: 'US' },
  { id: 'SKU-003', product: 'Mechanical Keyboard', category: 'Peripherals', stock: 28, region: 'EU' },
]

function labelFor(field: string) {
  return field.replaceAll('_', ' ').replace(/^\w/, (letter) => letter.toUpperCase())
}

const model = localModel({ data: rows })

export default function App() {
  const fields = React.useMemo(() => {
    const seen = new Set<string>()
    const next: string[] = []

    for (const row of rows) {
      for (const field of Object.keys(row)) {
        if (field !== 'id' && !seen.has(field)) {
          seen.add(field)
          next.push(field)
        }
      }
    }

    return next
  }, [])

  return (
    <DataTable className="rounded-xl" computeRowKey={({ data }) => data.id} model={model} style={{ height: 300 }}>
      {fields.map((field) => (
        <DataTableColumn field={field} key={field}>
          <DataTableColumnHeader>{labelFor(field)}</DataTableColumnHeader>
          <DataTableCell>{({ cellValue }) => String(cellValue ?? '')}</DataTableCell>
        </DataTableColumn>
      ))}
    </DataTable>
  )
}
```

## Model result discovery

If fields can change after filtering, grouping, permissions, or a remote fetch, subscribe to the
model result and derive the visible field list from the normalized `DataResult`.

```tsx
import type { DataModelHandle, DataResult } from '@virtuoso.dev/data-table'

function fieldsFromResult(result: DataResult<Record<string, unknown>>) {
  const fields: string[] = []
  const seen = new Set<string>()

  for (const row of result.data) {
    if (!row) {
      continue
    }

    for (const field of Object.keys(row)) {
      if (field !== 'id' && !seen.has(field)) {
        seen.add(field)
        fields.push(field)
      }
    }
  }

  return fields
}

function useDiscoveredFields(model: DataModelHandle<Record<string, unknown>>) {
  const [fields, setFields] = React.useState<string[]>([])

  React.useEffect(() => {
    return model.subscribe((message) => {
      if (message.type === 'result') {
        setFields(fieldsFromResult(message.payload as DataResult<Record<string, unknown>>))
      }
    })
  }, [model])

  return fields
}
```

This is useful when remote responses can expose different fields for different saved views, tenants,
or permission scopes. For ordinary remote tables, prefer explicit schema metadata so the header is
known before the first page resolves.

## Remote placeholders

Remote models can render placeholder rows while the first request is pending. If you infer fields
from model results, make sure placeholders contain the same fields as real rows or keep a separate
schema list from the request metadata.

```typescript
const loadingProduct = {
  id: 'loading',
  product: '',
  category: '',
  stock: null,
}

const model = remoteModel({
  fetch: fetchProducts,
  pageSize: 50,
  placeholder: loadingProduct,
})
```

If the backend can return zero rows, do not rely on row discovery alone. Use schema metadata or a
fallback field list so the table can still render headers and empty-state controls.

## Practical rules

- Prefer an explicit schema from your backend when one exists.
- Filter out technical fields such as internal IDs when they should not become visible columns.
- Keep `field` stable across datasets so column persistence can restore matching width, order, and
  visibility state.
- Keep generated field order deterministic. Do not let a later page reorder existing columns.
- If rows can include group labels, skip group-only fields or provide a `GroupHeaderCell` that handles
  them separately.
