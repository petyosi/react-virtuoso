This document captures the implemented slot-based composition system for column headers. The goal is to let users combine reusable header behaviors (reorder, sort, filter, resize) per-column, while keeping full control over header content.

## Implementation

- Slot registrar components live in `/Users/petyo/w/virtuoso/react-virtuoso/packages/data-table/src/columns/header-slots/slots.tsx`.
- The slot-aware `ColumnHeader` and `ColumnHeaderRenderer` live in `/Users/petyo/w/virtuoso/react-virtuoso/packages/data-table/src/columns/ColumnHeader.tsx`.
- The shadcn wrapper mirrors the same slot contract in `/Users/petyo/w/virtuoso/react-virtuoso/apps/virtuoso.dev/registry/new-york/data-table/data-table.tsx`.

Implemented runtime contract:

- Slot declarations are React element children.
- Header content is the single render-function child.
- Plain text and number children are not supported in slot mode.
- React element children are permissive so wrapped slot registrars can mount.
- Header layout belongs to `ColumnHeader` / `DataTableColumnHeader`, not the behavior slot components.

## Motivation

Today, customizing a column header means providing a single `component` or render function to `ColumnHeader`. That component is responsible for the entire header content -- label, drag handles, sort icons, resize affordances, everything. This creates two problems:

1. **Behaviors can't be mixed and matched.** To get reorder _and_ sort on a header, the user must build a single monolithic component that does both.
2. **Content can't be customized independently.** The behavior logic (drag handlers, sort state) is tangled with the label rendering, so adding a bold label to a sortable header means forking the entire sort component.

Existing features like `reorderColumns$` and `resizeColumn$` solve the _state_ side (they're reactive streams that any code can publish to); the slot chassis provides the compositional model for the _UI_ side.

## Design goals

- Per-column opt-in to behaviors: one column might be sortable + resizable, another might only have a reorder handle, a third might have no interactive behavior at all.
- Content customization orthogonal to behaviors: the label render function works the same regardless of which behaviors are active.
- Flat DOM: no wrapper-per-behavior nesting. All behaviors render as siblings inside the library's outer `<div>`.
- Backward compatible: existing `<ColumnHeader component={...} />` and `<ColumnHeader>{renderFn}</ColumnHeader>` continue to work unchanged.
- Fits the shadcn registry model: behaviors ship as copy-and-own components.

## Slot positions

The column header cell has a library-owned outer `<div>` (carries the `ResizeObserver` ref, width overrides, `data-column-key`). Inside that div, content is laid out in named positions:

```text
+---------------------------------------------------------------+
|  [start]   [content ----------- flex:1]   [end]   |edge|      |
|                                                    |    |      |
|  ................ overlay ........................  |    |      |
+---------------------------------------------------------------+
```

- **start** -- before content, in flow. Reorder grip, selection checkbox.
- **content** -- the label. Provided by render function or `component` prop. Gets `flex: 1`.
- **end** -- after content, in flow. Sort icon, filter button, column menu.
- **edge** -- `position: absolute; right: 0; top: 0; bottom: 0`. Resize handle.
- **overlay** -- `position: absolute; inset: 0; pointer-events: none`. Drop indicators, selection highlight.

All positions are siblings inside the library div. The div has `position: relative` so edge/overlay positions resolve correctly.

## Proposed API

### Declaring slots

Slot spec components are children of `ColumnHeader`. Each accepts `component` or `children` render function, mirroring `ColumnHeader`'s own API:

```tsx
<Column field="name">
  <ColumnHeader>
    <HeaderStart component={ReorderGrip} />
    <HeaderEnd component={SortIndicator} />
    <HeaderEdge component={ResizeHandle} />
    <HeaderOverlay component={ReorderDropZone} />
    {({ column }) => <span className="font-bold">{column.field}</span>}
  </ColumnHeader>
  <Cell>{({ cellValue }) => String(cellValue)}</Cell>
</Column>
```

The single render function among the children is the **content**. Everything else is a slot spec. Plain text and number children are intentionally rejected in slot mode.

### Slot component with user props

Slot components can accept user-provided props alongside the library-provided render params. For example, a sort indicator needs the current direction and a callback:

```tsx
<HeaderEnd component={SortIndicator} direction="asc" onSort={() => toggleSort('name')} />
```

`HeaderEnd` passes extra props through to the component. When the component renders, it receives both the library params (`columnKey`, `column`, `columnState`, `headerRef`) and the user props (`direction`, `onSort`).

### Children render prop variant

Slots also accept a render function as children, for inline customization:

```tsx
<HeaderEnd>
  {({ columnKey }) => <button onClick={() => toggleSort(columnKey)}>{sortState[columnKey] === 'asc' ? '↑' : '↓'}</button>}
</HeaderEnd>
```

### Minimal columns

Columns without slot children work exactly as today:

```tsx
<Column field="actions">
  <ColumnHeader>{() => 'Actions'}</ColumnHeader>
  <Cell>{({ row }) => <ActionButtons row={row} />}</Cell>
</Column>
```

### Mixed behaviors per column

Different columns opt into different behaviors independently:

```tsx
{
  /* Sortable + resizable */
}
;<Column field="name">
  <ColumnHeader>
    <HeaderEnd component={SortIndicator} direction={sortDir} onSort={onSort} />
    <HeaderEdge component={ResizeHandle} />
    {({ column }) => column.field}
  </ColumnHeader>
  <Cell>...</Cell>
</Column>

{
  /* Reorderable only */
}
;<Column field="status">
  <ColumnHeader>
    <HeaderStart component={ReorderGrip} />
    <HeaderOverlay component={ReorderDropZone} />
    {({ column }) => column.field}
  </ColumnHeader>
  <Cell>...</Cell>
</Column>

{
  /* No interactive behaviors */
}
;<Column field="id">
  <ColumnHeader>{() => 'ID'}</ColumnHeader>
  <Cell>...</Cell>
</Column>
```

## How it works internally

### Slot spec components follow the existing registration pattern

`ColumnHeader`, `Cell`, and `Column` all follow the same pattern: mount, register into a reactive cell via `useLayoutEffect`, return null. Slot specs do the same. Each slot position gets its own registry:

```tsx
const { cell$: headerStartEntries$, register$: headerStartRegister$ } = createRegistryCell<HeaderSlotEntry>()
const { cell$: headerEndEntries$, register$: headerEndRegister$ } = createRegistryCell<HeaderSlotEntry>()
const { cell$: headerEdgeEntries$, register$: headerEdgeRegister$ } = createRegistryCell<HeaderSlotEntry>()
const { cell$: headerOverlayEntries$, register$: headerOverlayRegister$ } = createRegistryCell<HeaderSlotEntry>()
```

A slot entry stores the column ID (from `useColumnId()`), the renderer, and any pass-through props:

```tsx
interface HeaderSlotEntry {
  columnId: string
  type: 'function' | 'component'
  renderer: ColumnHeaderRenderFunction | ColumnHeaderCustomComponent
  extraProps?: Record<string, unknown>
}
```

A slot spec component (e.g., `HeaderStart`) registers into its position's registry:

```tsx
function HeaderStart({ component, children, ...extraProps }) {
  const colId = useColumnId()
  const slotId = useId()
  const register = usePublisher(headerStartRegister$)

  const renderer = children ?? component
  const rendererType = children ? 'function' : 'component'

  useLayoutEffect(() => {
    register({
      type: 'add',
      id: slotId,
      value: { columnId: colId, type: rendererType, renderer, extraProps },
    })
    return () => register({ type: 'remove', id: slotId })
  }, [register, slotId, colId, renderer, rendererType, extraProps])

  return null
}
```

### `ColumnHeader` renders slot children

Today `ColumnHeader` returns `null`. With slot children, it renders them so they can mount and register:

```tsx
function ColumnHeader(props) {
  // ... existing content registration logic (unchanged) ...

  // NEW: if children include React elements (slot specs), render them
  // They mount, register via useLayoutEffect, and return null
  return slotChildren.length > 0 ? <>{slotChildren}</> : null
}
```

`ColumnHeader` uses a permissive runtime contract: one function child becomes the content renderer, React element children are allowed to mount so wrapped slot registrars work, and plain text / number children are rejected in slot mode.

### `ColumnHeaderRenderer` reads slot registries

`ColumnHeaderRenderer` subscribes to the slot registry cells, filters for the current column, and renders:

```tsx
function ColumnHeaderRenderer({ columnKey, column, columnState, renderer, rendererType, ... }) {
  const ref = useResizeObserver('border-box')
  // ... existing width override logic ...

  // Read slot registries
  const allStartSlots = useCellValue(headerStartEntries$)
  const allEndSlots = useCellValue(headerEndEntries$)
  const allEdgeSlots = useCellValue(headerEdgeEntries$)
  const allOverlaySlots = useCellValue(headerOverlayEntries$)

  // Filter to this column
  const startSlots = filterByColumn(allStartSlots, columnKey)
  const endSlots = filterByColumn(allEndSlots, columnKey)
  const edgeSlots = filterByColumn(allEdgeEntries, columnKey)
  const overlaySlots = filterByColumn(allOverlayEntries, columnKey)

  const params = { columnKey, column, columnState, overlaidByScrollbar, headerRef: ref }
  const hasSlots = startSlots.length + endSlots.length + edgeSlots.length + overlaySlots.length > 0

  return (
    <div ref={ref} className={className}
         data-table-element-role="column-header" data-column-key={columnKey}
         style={style}>
      {hasSlots ? (
        <>
          {renderSlotEntries(overlaySlots, params)}
          {renderSlotEntries(startSlots, params)}
          {renderContent(renderer, rendererType, params)}
          {renderSlotEntries(endSlots, params)}
          {renderSlotEntries(edgeSlots, params)}
        </>
      ) : (
        renderContent(renderer, rendererType, params)
      )}
    </div>
  )
}
```

When a column has no registered slots, the render path is identical to today.

### `headerRef` -- how slot components interact with the cell element

Slot component renderers receive `headerRef` (a ref to the library's outer div) as part of their params. This is how behaviors attach event listeners to the cell without needing wrapper elements:

```tsx
interface HeaderSlotRenderParams extends ColumnHeaderRenderParams {
  headerRef: React.RefObject<HTMLDivElement>
}
```

For example, a reorder drop zone attaches native `dragover`/`drop` listeners to the cell element via `headerRef`:

```tsx
function ReorderDropZone({ columnKey, columnState, headerRef }) {
  const reorderColumns = usePublisher(reorderColumns$)
  const [dropPosition, setDropPosition] = useState(null)

  useEffect(() => {
    const el = headerRef.current
    if (!el) return

    const onDragOver = (ev) => {
      if (!dragSourceKey || dragSourceKey === columnKey) return
      if (dragSourceSticky !== columnState.sticky) return
      ev.preventDefault()
      const rect = el.getBoundingClientRect()
      setDropPosition(ev.clientX < rect.left + rect.width / 2 ? 'before' : 'after')
    }
    const onDrop = (ev) => {
      ev.preventDefault()
      if (dragSourceKey && dragSourceKey !== columnKey && dropPosition) {
        reorderColumns({ sourceKey: dragSourceKey, targetKey: columnKey, position: dropPosition })
      }
      dragSourceKey = null
      setDropPosition(null)
    }
    const onDragLeave = () => setDropPosition(null)

    el.addEventListener('dragover', onDragOver)
    el.addEventListener('drop', onDrop)
    el.addEventListener('dragleave', onDragLeave)
    return () => {
      el.removeEventListener('dragover', onDragOver)
      el.removeEventListener('drop', onDrop)
      el.removeEventListener('dragleave', onDragLeave)
    }
  }, [headerRef, columnKey, columnState.sticky, reorderColumns, dropPosition])

  if (!dropPosition) return null
  return (
    <div style={{ position: 'absolute', inset: '0', pointerEvents: 'none' }}>
      <div
        style={{
          position: 'absolute',
          top: 0,
          bottom: 0,
          width: 3,
          background: 'var(--color-primary)',
          [dropPosition === 'before' ? 'left' : 'right']: 0,
        }}
      />
    </div>
  )
}
```

This pattern (native listeners via ref) avoids event handler merging, ordering conflicts, and the need to funnel React props upward from children to the parent div.

## Worked example: full reorder implementation

Using this design, column reorder splits into two slot components:

```tsx
{
  /* Per-column declaration */
}
;<Column field="name">
  <ColumnHeader>
    <HeaderStart component={ReorderGrip} />
    <HeaderOverlay component={ReorderDropZone} />
    {({ column }) => column.field}
  </ColumnHeader>
  <Cell>{({ cellValue }) => String(cellValue)}</Cell>
</Column>
```

`ReorderGrip` (start slot) -- the draggable handle:

```tsx
function ReorderGrip({ columnKey, columnState }: HeaderSlotRenderParams) {
  return (
    <span
      draggable
      onDragStart={(ev) => {
        dragSourceKey = columnKey
        dragSourceSticky = columnState.sticky
        ev.dataTransfer.effectAllowed = 'move'
        ev.dataTransfer.setData('text/plain', columnKey)
        const header = ev.currentTarget.closest('[data-table-element-role="column-header"]')!
        const rect = header.getBoundingClientRect()
        ev.dataTransfer.setDragImage(header, ev.clientX - rect.left, ev.clientY - rect.top)
      }}
      onDragEnd={() => {
        dragSourceKey = null
        dragSourceSticky = undefined
      }}
      style={{ cursor: 'grab', marginRight: 8, userSelect: 'none', opacity: 0.5, fontSize: '14px' }}
    >
      ⠿
    </span>
  )
}
```

`ReorderDropZone` (overlay slot) -- drop target with visual indicator. Uses `headerRef` to attach `dragover`/`drop`/`dragleave` listeners to the cell element. See the `headerRef` section above for the full implementation.

Module-level drag state (shared between grip and drop zone):

```tsx
let dragSourceKey: string | null = null
let dragSourceSticky: string | undefined
```

The module-level singleton is necessary because `dataTransfer.getData()` returns `""` during `dragover` in Chrome and Safari.

## Worked example: resize handle

Resize maps to the edge slot:

```tsx
<Column field="name">
  <ColumnHeader>
    <HeaderEdge component={ResizeHandle} />
    Product Name
  </ColumnHeader>
  <Cell>...</Cell>
</Column>
```

`ResizeHandle` (edge slot):

```tsx
function ResizeHandle({ columnKey }: HeaderSlotRenderParams) {
  const resizeColumn = usePublisher(resizeColumn$)

  return (
    <div
      style={{
        position: 'absolute',
        right: 0,
        top: 0,
        bottom: 0,
        width: 4,
        cursor: 'col-resize',
      }}
      onPointerDown={(ev) => {
        ev.preventDefault()
        const startX = ev.clientX
        const cell = ev.currentTarget.parentElement!
        const startWidth = cell.getBoundingClientRect().width
        ev.currentTarget.setPointerCapture(ev.pointerId)

        const onMove = (e: PointerEvent) => {
          resizeColumn({ key: columnKey, width: Math.max(50, startWidth + e.clientX - startX) })
        }
        const onUp = () => {
          document.removeEventListener('pointermove', onMove)
          document.removeEventListener('pointerup', onUp)
        }
        document.addEventListener('pointermove', onMove)
        document.addEventListener('pointerup', onUp)
      }}
    />
  )
}
```

The handle is absolutely positioned at the cell's right edge. This works because it's a direct child of the library's outer div (which has `position: relative`), not nested inside wrapper divs.

## Worked example: sort indicator

Sort maps to the end slot with user-provided state:

```tsx
<Column field="price">
  <ColumnHeader>
    <HeaderEnd component={SortIndicator} direction={sortState.price} onSort={() => toggleSort('price')} />
    Price
  </ColumnHeader>
  <Cell>...</Cell>
</Column>
```

`SortIndicator` (end slot) receives both library params and user props:

```tsx
function SortIndicator({ direction, onSort, headerRef }: HeaderSlotRenderParams & { direction?: 'asc' | 'desc'; onSort: () => void }) {
  // Attach click-to-sort on the header cell
  useEffect(() => {
    const el = headerRef.current
    if (!el) return
    const onClick = (ev: MouseEvent) => {
      // Don't trigger sort when clicking other interactive elements
      if ((ev.target as HTMLElement).closest('[data-header-interactive]')) return
      onSort()
    }
    el.addEventListener('click', onClick)
    return () => el.removeEventListener('click', onClick)
  }, [headerRef, onSort])

  return (
    <span style={{ marginLeft: 4, opacity: direction ? 1 : 0.3 }} data-header-interactive>
      {direction === 'asc' ? '↑' : direction === 'desc' ? '↓' : '↕'}
    </span>
  )
}
```

The `data-header-interactive` attribute lets the click handler on the cell skip clicks that originate from other interactive slot elements (filter buttons, grip handles), preventing conflicts.

## Combining behaviors

Multiple behaviors compose naturally:

```tsx
<Column field="name">
  <ColumnHeader>
    <HeaderStart component={ReorderGrip} />
    <HeaderOverlay component={ReorderDropZone} />
    <HeaderEnd component={SortIndicator} direction={sortState.name} onSort={() => toggleSort('name')} />
    <HeaderEnd component={FilterButton} options={filterOptions.name} value={filters.name} onChange={setNameFilter} />
    <HeaderEdge component={ResizeHandle} />
    {({ column }) => <span className="font-bold">{column.field}</span>}
  </ColumnHeader>
  <Cell>...</Cell>
</Column>
```

The rendered DOM is flat:

```html
<div data-table-element-role="column-header" style="display:flex; align-items:center; position:relative">
  <!-- overlay: drop indicator (only during drag) -->
  <!-- start: grip ⠿ -->
  <!-- content: bold label -->
  <!-- end: sort icon ↑ -->
  <!-- end: filter button -->
  <!-- edge: resize handle (absolute, right edge) -->
</div>
```

No nesting, no wrapper divs per behavior. Event conflicts are avoided because:

- Reorder drag and sort click use different event types (`dragover`/`drop` vs `click`).
- The resize handle captures pointer events on its own element and doesn't bubble.
- The filter button uses `data-header-interactive` to exclude itself from the sort click handler.

## Shadcn registry packaging

The shadcn registry would ship:

1. Individual slot components (`ReorderGrip`, `ReorderDropZone`, `SortIndicator`, `FilterButton`, `ResizeHandle`) as copy-and-own source files.
2. A pre-composed `DataTableColumnHeader` convenience wrapper for common combinations:

```tsx
// Pre-composed: handles common props, assembles slots internally
<DataTableColumnHeader sortable resizable>
  Product Name
</DataTableColumnHeader>

// Equivalent manual composition:
<DataTableColumnHeader>
  <HeaderEnd component={SortIndicator} direction={...} onSort={...} />
  <HeaderEdge component={ResizeHandle} />
  Product Name
</DataTableColumnHeader>
```

Users who need full control use the slot components directly. Users who want convenience use the pre-composed wrapper. Both use the same underlying mechanism.

## Known issues and open questions

### Slot registry performance

Each slot position has its own reactive cell. Every `ColumnHeaderRenderer` subscribes to all slot cells and filters for its column. When any column's slot registration changes, all headers re-evaluate the filter. With typical column counts (10--50) and 1--3 slots per column, the filtering is cheap (small map iteration + string comparison), but it's worth profiling at scale.

Alternative: a single `headerSlots$` cell with position stored in the value. Same filter cost, fewer subscriptions.

### Distinguishing slot children from content children

`ColumnHeader` needs to tell apart slot elements (`HeaderStart`, `HeaderEnd`, etc.) from content (render functions, strings, React elements used as labels). The proposed approach: slot components carry a static marker (`HeaderStart.__headerSlot = true`). Anything without the marker is content. This is robust against HOC wrapping if the marker is preserved, but fragile if a user wraps a slot in `React.memo` or a custom HOC without forwarding the static property.

### `useResizeObserver` ref sharing

`ColumnHeaderRenderer` currently gets a ref callback from `useResizeObserver`. Slot components need a `RefObject` (for `addEventListener`). The `useResizeObserver` hook would need to also populate a `RefObject` that's passed to slots as `headerRef`. Minor plumbing change.

### Multiple instances of the same position

Two `<HeaderEnd>` components both register in the end position. Both render. Their order follows `useId()` generation, which mirrors JSX child order. This is predictable but implicit -- the user controls order by arranging children.

### User props pass-through typing

`HeaderEnd` needs to forward user-provided props (like `direction`, `onSort`) to the slot component. This makes the `HeaderEnd` prop type either generic over the component's props or loosely typed with `Record<string, unknown>`. A generic approach is more type-safe:

```tsx
function HeaderEnd<P>(props: { component: React.ComponentType<HeaderSlotRenderParams & P> } & P): null
```

But this requires TypeScript inference to extract `P` from the component type, which can be fragile with complex component signatures.

### Features that don't map cleanly to slots

**Column grouping drag.** A second drag behavior alongside reorder. Both use `dragstart`/`dragover`/`drop`. They'd need mutually exclusive activation (different handle, modifier key, or long press). Not a slot problem per se, but composing two drag behaviors on the same header is inherently complex.

**Inline header rename.** Needs to replace the content area temporarily (switching between display and edit mode). The content slot would need to support conditional rendering or the rename behavior would overlay the content via the overlay slot with an input field. Workable but less natural than the other behaviors.

### Backward compatibility

The design is fully backward compatible:

- `<ColumnHeader component={MyHeader} />` -- no slot children. `ColumnHeader` returns null. `ColumnHeaderRenderer` renders exactly as today.
- `<ColumnHeader>{(params) => params.column.field}</ColumnHeader>` -- render function, no slots. Same as today.
- `<ColumnHeader component={MyContent}><HeaderStart component={Grip} /></ColumnHeader>` -- new mode. `MyContent` fills the content area, slots surround it.

When slot children are present, `component` on `ColumnHeader` means "the content area renderer." When no slot children are present, it means "the entire header content" -- which is the same thing, since there are no surrounding slots.
