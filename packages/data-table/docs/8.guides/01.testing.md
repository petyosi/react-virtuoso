Virtualized tables measure the viewport before they decide which rows to render. In browser tests
that is usually fine. In component tests, layout can be missing, delayed, or different from the real
browser. The testing context gives the table known dimensions so assertions do not depend on
measurement timing.

Use it when a test needs to assert which rows are visible, whether scrolling loads the next range,
or whether a loading/empty state appears in a predictable viewport.

## Fixed-size test table

```tsx live
import { localModel, VirtuosoDataTableTestingContext } from '@virtuoso.dev/data-table'
import { DataTable, DataTableCell, DataTableColumn, DataTableColumnHeader } from '@/components/ui/data-table'

const rows = Array.from({ length: 20 }, (_, index) => ({
  id: `SKU-${String(index + 1).padStart(3, '0')}`,
  name: `Product ${index + 1}`,
}))

const model = localModel({ data: rows })

export default function App() {
  return (
    <VirtuosoDataTableTestingContext.Provider value={{ itemHeight: 40, viewportHeight: 240 }}>
      <DataTable className="rounded-xl" model={model} style={{ height: 240 }}>
        <DataTableColumn field="name">
          <DataTableColumnHeader>Product</DataTableColumnHeader>
          <DataTableCell>{({ cellValue }) => String(cellValue)}</DataTableCell>
        </DataTableColumn>
      </DataTable>
    </VirtuosoDataTableTestingContext.Provider>
  )
}
```

`itemHeight` is the row height the table should assume in the test. `viewportHeight` is the visible
vertical space. In the example above, a 240px viewport with 40px rows gives the table room for about
six rows before overscan.

The table still needs a real height in its own layout. Keep the provider dimensions and the table
height aligned unless the test intentionally covers a mismatch.

## What to assert

Prefer assertions about behavior, not the exact number of DOM nodes. Virtualization can render a few
extra rows for overscan, sticky headers, or measurement buffers.

Good browser-test assertions:

- a row near the top is visible before scrolling
- a target row becomes visible after `scrollToRow$` or user scrolling
- an empty placeholder is visible when the model has no rows
- a remote loading overlay appears while a request is pending

For component tests, wrap the table once in a shared helper:

```tsx
function renderTestTable(children: React.ReactNode) {
  return render(
    <VirtuosoDataTableTestingContext.Provider value={{ itemHeight: 40, viewportHeight: 240 }}>
      {children}
    </VirtuosoDataTableTestingContext.Provider>
  )
}
```

## Browser vs. component tests

- Use deterministic heights in tests so assertions do not depend on browser layout timing.
- In browser tests, prefer the real layout when you are testing sticky columns, resize handles, drag
  interactions, or custom scroll parents.
- In component tests, use `VirtuosoDataTableTestingContext` for visible-row and loading-state
  assertions.
- Keep data generation simple. You are usually validating behavior, not visual polish.
