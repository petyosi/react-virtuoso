The biggest performance wins usually come from controlling what enters the viewport and keeping row
identity stable. Start with the defaults, then tune only when a specific table shows a specific
symptom.

## Tuned viewport example

```tsx live
import { DataTable, DataTableCell, DataTableColumn, DataTableColumnHeader } from '@/components/ui/data-table'
import { localModel } from '@virtuoso.dev/data-table'

const rows = Array.from({ length: 240 }, (_, index) => ({
  id: `SKU-${String(index + 1).padStart(3, '0')}`,
  name: `Product ${index + 1}`,
  region: ['US', 'EU', 'APAC'][index % 3]!,
  status: ['In stock', 'Low stock', 'Backorder'][index % 3]!,
}))

const model = localModel({ data: rows })

export default function App() {
  return (
    <DataTable
      className="rounded-xl"
      columnOverscanCount={1}
      computeRowKey={({ data }) => ('id' in data ? data.id : String(data))}
      model={model}
      increaseViewportBy={120}
      style={{ height: 360 }}
    >
      <DataTableColumn field="name">
        <DataTableColumnHeader>Product</DataTableColumnHeader>
        <DataTableCell>{({ cellValue }) => String(cellValue)}</DataTableCell>
      </DataTableColumn>
      <DataTableColumn field="region">
        <DataTableColumnHeader>Region</DataTableColumnHeader>
        <DataTableCell>{({ cellValue }) => String(cellValue)}</DataTableCell>
      </DataTableColumn>
      <DataTableColumn field="status">
        <DataTableColumnHeader>Status</DataTableColumnHeader>
        <DataTableCell>{({ cellValue }) => String(cellValue)}</DataTableCell>
      </DataTableColumn>
    </DataTable>
  )
}
```

## What each setting does

- `computeRowKey` preserves React row identity when rows move because of sorting, filtering,
  grouping, prepends, appends, or replacement.
- `increaseViewportBy` renders extra vertical pixels before and after the viewport. Use it when fast
  scrolling reveals blank space or when expensive media should start loading before it enters view.
- `columnOverscanCount` renders extra columns on each side of the horizontal viewport. Use it when a
  very wide table feels empty or jumpy during horizontal scrolling.

Overscan has a cost. More overscan means more mounted cells and more render work. Increase it in
small steps and keep the value local to the table that needs it.

## Symptoms and fixes

- Rows remount or lose local state after sorting: add `computeRowKey`.
- Horizontal scrolling shows empty cells briefly: raise `columnOverscanCount` by one.
- Vertical scrolling exposes blank space around heavy rows: add a modest `increaseViewportBy`.
- Typing in filters feels slow: move expensive filtering into a local model pipeline or to the
  backend.
- Cell renderers are expensive: memoize shared formatters outside the renderer and avoid creating
  large objects per cell.

Use [Inspecting Row Re-renders](/data-table/guides/debug-instrumentation/) when you need to
confirm which rows are rendering during a scroll or update.
